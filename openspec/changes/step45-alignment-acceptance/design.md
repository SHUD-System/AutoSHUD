## Scope

This change defines the implementation fixture for AutoSHUD Step4 and Step5
alignment. It is intentionally narrower than full scientific model validation:
the target is deterministic execution contracts, output isolation, and
post-processing smoke coverage.

## Decisions

- Step4 must be configurable from project config, environment variables, or
  explicit harness options instead of hardcoding a `git clone` path. Rationale:
  tests and normal users should not need network or SSH availability just to
  verify AutoSHUD orchestration.
- Real-case Step4 execution must start from an explicitly configured local SHUD
  source directory and must run `make clean` followed by `make shud` for each
  case before that case is executed. Rationale: case runs should not silently
  reuse a stale binary built for a previous case or environment.
- The acceptance must fail if local SHUD source, `make clean`, `make shud`,
  solver execution, or required real solver outputs are unavailable. Rationale:
  Step4-5 validation must not silently downgrade to non-real solver evidence.
- The freshly built `shud` executable must be copied/staged into the
  case-owned run directory before execution. Rationale: each case needs an
  auditable solver binary boundary and isolated logs/outputs.
- The Step4-5 acceptance in this workspace must use the local SHUD source
  checkout. Rationale: the validation target is real AutoSHUD Step4-5 behavior
  against the actual solver build/run path.
- Step5 should be refactored into callable helpers before or alongside script
  alignment. Rationale: `Step5_ResultVisualization.R` and
  `Step5_WaterBalance.R` need automated tests that assert structured outputs
  rather than depending on interactive console inspection.
- Step5 tests should use outputs generated by the real Step4 solver smoke run.
  Rationale: Step5 should validate readers, water-balance arithmetic, required
  variables, and plot publication against the actual Step4 output contract.
- Step5 output validation must require these solver variables:
  `eleysurf`, `eleyunsat`, `eleygw`, `elevprcp`, `elevetp`, `elevinfil`,
  `elevrech`, `eleqsurf`, `elevettr`, `elevetic`, `elevetev`,
  `rivystage`, `rivqdown`, `rivqsub`, and `rivqsurf`. Rationale: the smoke
  check must not pass with partial or non-real solver outputs.
- The Step4-5 harness must first assert that required Step3 model-input files
  exist before building/running SHUD. Rationale: Step4 failures should not hide
  missing Step3 prerequisites.
- The acceptance must write a Chinese structured test report under a
  harness-owned artifact/report directory. Rationale: long-running real solver
  evidence needs a durable human-readable record, while public PR/issue text
  should not expose private absolute paths.
- Runtime controls must be explicit: build and solver working directories,
  timeout, single-process/thread environment unless explicitly configured, and
  retained failure logs. Rationale: SHUD runs can hang or produce large logs;
  acceptance needs stable bounds and actionable diagnostics.
- Generated logs, solver outputs, figures, and build products must stay under a
  temporary run root or configured ignored output directory. Rationale: Step4-5
  are output-heavy and must not mutate `Example/` or tracked fixtures.
- Step5 GIS/CRS risk is limited to verifying that spatial or figure outputs
  produced by Step5 stay under the configured analysis/output directory and do
  not mutate Step3 GIS sidecars. Rationale: this issue is not a CRS migration,
  but Step5 must not publish GIS/plot side effects outside the run root.

## Risk Triage

Issue type: feature/test hardening
Project profile: AutoSHUD
Blast radius: high
Fixture level: expanded

Why:

- Step4 invokes an external executable and historically performs network/build
  side effects.
- Step5 consumes solver outputs and computes water-balance summaries from many
  output variables.
- File IO, working directory, generated figures, and logs are central to the
  user-facing workflow.
- The change must not weaken the existing Step1-3 acceptance or ERA5/forcing
  hardening tests.

## Risk Packs Considered

- Public API / CLI / script entry: selected - Step4/Step5 scripts are
  user-facing workflow entrypoints.
- Config / project setup: selected - SHUD source directory, solver executable,
  run directory, model input path, and output path must be explicit.
- File IO / path safety / overwrite: selected - solver logs, copied
  executables, result files, and figures must not pollute tracked examples.
- Schema / columns / units / field names: selected - Step5 depends on SHUD
  output variable names, model input tables, and time-series shapes.
- Geospatial / CRS / shapefile sidecars: selected - Step5 visualization may
  read mesh/river GIS or convert mesh outputs.
- Time series / forcing / temporal boundaries: selected - Step5 water balance
  must respect time index length, solver output step, and simulation window.
- Numerical stability / conservation / NaN: selected - Step5 smoke checks must
  reject missing, non-finite, or dimensionally inconsistent output values; no
  hydrologic accuracy claim.
- Solver runtime / performance / threading: selected - Step4 must support real
  per-case SHUD rebuild/run in this workspace without uncontrolled parallelism.
- Resource limits / large input / discovery: selected - Step4/5 tests must use
  small cases and must not clone solver sources by default; real builds use an
  existing local SHUD source checkout.
- Legacy compatibility / examples: selected - existing example configs and
  Step1-3 acceptance must continue to work.
- Error handling / rollback / partial outputs: selected - failed solver runs
  must leave clear logs and must not publish misleading Step5 summaries.
- Release / packaging / dependency compatibility: selected - no new heavy
  runtime dependency should be added only for tests; dependency metadata must
  not change unless strictly required and reviewed.
- Documentation / migration notes: selected - users need explicit instructions
  for the local SHUD source requirement and real per-case rebuild/run behavior.

## Required Evidence

- OpenSpec validation: `openspec validate step45-alignment-acceptance --strict --no-interactive`.
- Step4 build/run ordering evidence: real build logs show `make clean` and
  `make shud` in that order for each selected case before solver execution.
- Step4 command construction evidence: the real solver receives the expected
  project name and runs from the intended working directory.
- Step4 no-solver failure test: absence of solver configuration fails clearly
  before clone/build/execution side effects.
- Step4 output isolation test: build logs, solver outputs, copied executable
  files, and temporary build artifacts stay under the harness run root.
- Step4 real-solver evidence: in this workspace, local SHUD source is present,
  so the primary acceptance must rebuild and run real SHUD for each selected
  case with `make clean && make shud`.
- Step4 runtime controls evidence: build/run logs include working directory,
  command, timeout, relevant environment settings, exit status, and retained
  failure logs.
- Step4 no-downgrade evidence: missing local source, failed build, failed
  solver run, or missing required solver outputs fails the acceptance rather
  than substituting non-real outputs.
- Step3 prerequisite evidence: before Step4, the harness verifies the generated
  model-input directory contains `.sp.mesh`, `.sp.riv`, `.sp.att`,
  `.sp.rivseg`, `.cfg.para`, `.cfg.calib`, `.cfg.ic`, and `.tsd.forc`.
- Step5 result-reader test: real Step4 solver outputs can be read for the
  required variables without missing-column or dimension errors.
- Step5 failure tests: missing variable, dimension mismatch, and non-finite
  output values fail with actionable errors before writing misleading summary
  output.
- Step5 water-balance smoke test: the summary is finite, structured, and
  includes precipitation, evapotranspiration components, discharge components,
  storage deltas, and mass-balance residual.
- Step5 plot-output test: any generated figure is written to the configured
  analysis/output directory and not to repository root.
- Geospatial/CRS sidecar evidence: Step5 spatial/plot outputs, if produced,
  are non-empty, remain under the analysis/output directory, and do not mutate
  Step3 GIS sidecars.
- Dependency evidence: inspect diff to confirm no new heavy package dependency
  or dependency metadata churn.
- Regression commands: existing `Rscript tests/test-9035800-step123-acceptance.R`,
  `Rscript tests/test-step-forcing-hardening.R`, and
  `Rscript tests/test-era5-forcing.R` pass.
- Chinese structured report: generated report records commands, sanitized input
  and output paths, SHUD source commit/status, build/run exit codes, log
  locations, Step5 summary fields, pass/fail status, and known limits.

## Review Focus

- Does the Step4 design remove implicit network/SSH/build behavior from the
  default acceptance path?
- Does the acceptance run real local SHUD, rebuilding for each case before
  execution?
- Are Step5 checks strong enough to catch missing variables, wrong time
  dimensions, non-finite values, and incorrect output paths?
- Are all generated Step4-5 artifacts kept out of tracked examples and source
  fixture paths?
- Are `Example/9035800`, `testdata/9035800`, and repository-root plot artifacts
  explicitly protected in success and failure paths?
- Is the real solver path still available for maintainers who explicitly
  configure a local SHUD source directory, and does it rebuild for each case?
